title: "Design philosophy: single-file algorithms in JaxAHT"
description: "Explore how JaxAHT uses single-file algorithms for fast iteration, shared agent interfaces for cross-algorithm reuse, and JAX JIT for speed."

JaxAHT is built around two tensions that every research framework must navigate: keeping algorithms easy to understand and modify, while still providing enough shared infrastructure to avoid duplicating boilerplate across every implementation. The design choices described on this page reflect how JaxAHT resolves those tensions.

## Single-file algorithms

JaxAHT takes direct inspiration from [JaxMARL](https://github.com/FLAIROx/JaxMARL) and [CleanRL](https://github.com/vwxyzjn/cleanrl): each algorithm lives in a single self-contained file. The complete training logic — network definitions, loss functions, update steps, logging — sits in one place with no hidden dependencies spread across a deep class hierarchy.

This makes it straightforward to:

- Read any algorithm end-to-end without jumping between files
- Copy and modify a single file to prototype a new method
- Understand exactly what is happening at every step of training

The tradeoff is intentional code duplication between algorithms. JaxAHT accepts this cost because algorithm-specific logic is more important to expose clearly than DRY compliance.

<Note>
The single-file paradigm means you should treat each algorithm file as the authoritative source of truth for that algorithm. If you want to change how LIAM updates its policy, edit `ego_agent_training/liam_ego.py` — not a shared base class.
</Note>

## What is modularized

While algorithm logic stays in single files, three concerns are modularized because they genuinely need to be shared across algorithms:

**Agents** (`agents/`): Actor-critic network architectures, heuristic agents, and the `AgentPolicy` abstract base class that every policy must implement. Modularizing agents means a policy trained by IPPO can be loaded directly into an ego agent training algorithm without any adapter code.

**Populations** (`agents/population_interface.py`): The `AgentPopulation` class wraps a policy class and a population size, and provides batched action sampling via `jax.vmap`. Any algorithm that trains or evaluates against a set of partners uses this interface.

**Environments** (`envs/`): The `BaseEnv` abstract class, the `LogWrapper`, and environment-specific wrappers for LBF and Overcooked. All environments expose the same `step`, `reset`, `get_avail_actions`, `observation_space`, and `action_space` interface.

Everything else — how training steps are structured, how gradients flow, how partner agents are sampled during training — lives inside the individual algorithm files.

## The shared agent interface

The `AgentPolicy` abstract base class in `agents/agent_interface.py` is the linchpin of cross-algorithm compatibility. Every policy — whether trained by PPO, LIAM, MeLIBA, or IPPO — must implement `get_action`:

```python
@abc.abstractmethod
@partial(jax.jit, static_argnums=(0,))
def get_action(self, params, obs, done, avail_actions, hstate, rng,
               aux_obs=None, env_state=None, test_mode=False) -> Tuple[int, chex.Array]:
    """
    Only computes an action given an observation, done flag, available actions,
    hidden state, and random key.

    Returns:
        Tuple[int, chex.Array]: A tuple containing the action and the new hidden state.
    """
    pass
```

The interface also provides `get_action_value_policy` (used during training when you need value estimates and the full policy distribution, not just a sampled action), `init_hstate` (to initialize recurrent state before an episode), and `init_params`.

Because every algorithm produces policies that satisfy this interface, you can freely mix and match: take teammates generated by FCP, pass them to PPO Ego, then evaluate the resulting ego agent against heuristic agents — all without writing glue code.

## JAX and JIT compilation

JaxAHT uses [JAX](https://github.com/google/jax) throughout. The key benefit for AHT research is that `jax.jit` can compile the **entire training loop** — environment steps, policy forward passes, gradient updates, and population sampling — into a single optimized XLA computation. This eliminates Python interpreter overhead and achieves GPU/TPU throughput that would be impossible with PyTorch-style step-by-step execution.

The `@partial(jax.jit, static_argnums=(0,))` decorator on `get_action` and `get_action_value_policy` tells JAX to treat `self` (the policy object) as a static compile-time constant. This allows the JIT-compiled function to specialize on the specific policy architecture.

Episode rollouts are implemented with `jax.lax.scan`, which unrolls a fixed-length loop efficiently without Python-level iteration:

```python
final_carry, _ = jax.lax.scan(
    scan_step, init_carry, None, length=max_episode_steps)
```

Multiple episodes are then run in parallel with `jax.vmap`:

```python
vmap_run_single_episode = jax.jit(jax.vmap(
    lambda ep_rng: run_single_episode(
        ep_rng, env, agent_0_param, agent_0_policy,
        agent_1_param, agent_1_policy, max_episode_steps,
        agent_0_test_mode, agent_1_test_mode
    )
))
```

## Hydra configuration

JaxAHT uses [Hydra](https://hydra.cc) to manage algorithm and task configurations. Each algorithm directory contains a `configs/` subtree with three subdirectories:

- `configs/algorithm/`: One config per algorithm–task combination
- `configs/task/`: Environment-specific settings (grid size, number of agents, etc.)
- `configs/hydra/`: Hydra runtime settings (output directory, sweeper config)

Hydra merges these into a single master config at runtime. You can select a combination on the command line and override any individual value without editing any file:

```bash
# Run FCP on LBF with the default config
python teammate_generation/run.py \
  task=lbf/lbf_7x7_nolevels \
  algorithm=fcp/lbf/lbf_7x7_nolevels

# Override a single hyperparameter
python teammate_generation/run.py \
  task=lbf/lbf_7x7_nolevels \
  algorithm=fcp/lbf/lbf_7x7_nolevels \
  algorithm.TOTAL_TIMESTEPS=1e5
```

Results are logged to a local `results/` directory and to [Weights & Biases](https://wandb.ai). You can toggle W&B logging from the master config without touching the algorithm code.

## Project directory structure

```
jax-aht/
├── agents/                   # Agent interfaces, architectures, heuristics
│   ├── agent_interface.py    # AgentPolicy abstract base class
│   ├── population_interface.py  # AgentPopulation class
│   └── ...                   # MLP/RNN actor-critics, heuristic agents
│
├── common/                   # Shared utilities used across algorithm types
│   └── run_episodes.py       # Vectorized episode rollout helpers
│
├── envs/                     # Environment wrappers
│   ├── base_env.py           # BaseEnv abstract class
│   ├── log_wrapper.py        # Episode return/length logging
│   └── lbf/, overcooked/     # Environment-specific wrappers
│
├── ego_agent_training/       # PPO Ego, LIAM, MeLIBA
│   ├── run.py                # Entry point for all ego training methods
│   └── configs/              # Algorithm and task configs
│
├── marl/                     # IPPO
│   ├── run.py
│   └── configs/
│
├── open_ended_training/      # ROTATE, PAIRED, Open-Ended Minimax
│   ├── run.py
│   └── configs/
│
├── teammate_generation/      # FCP, BRDiv, LBRDiv, CoMeDi
│   ├── run.py
│   └── configs/
│
├── evaluation/               # Heldout evaluation and visualization scripts
└── tests/                    # Development test scripts
```

<Note>
Every algorithm directory uses `run.py` as its single entry point. You always invoke experiments through `run.py`, never by calling individual algorithm files directly. Hydra intercepts the call and assembles the correct config before passing control to the algorithm.
</Note>
